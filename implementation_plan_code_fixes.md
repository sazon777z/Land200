# Implementation Plan: Code Fixes

> Assumption: отдельный файл с Code Review report в репозитории не найден; план ниже собран по фактическим замечаниям, которые читаются из текущей кодовой базы `Land_Cruiser`.

## 1. Priority 1: Critical & Stability (High Impact)
(Исправление багов, утечек памяти, зависаний)

- [ ] Шаг 1: Защитить общий runtime-state от гонок между `TaskNetwork`, `TaskDisplay` и `TaskWeb`.
  Категория: Работа с памятью.
  Что будет изменено: в `NetworkManager` вынести изменяемые данные состояния (`currentTemp`, `currentCondition`, `currentIcon`, `weatherCity`, alarm settings) в единый snapshot/state-struct; добавить отдельный mutex или критическую секцию на чтение/запись; убрать небезопасные конкурентные операции со `String`.
  Что это решит: снизит риск heap corruption, случайных перезагрузок и получения частично обновленных данных в web/UI.
  Риски: изменятся сигнатуры getter/setter-методов; потребуется синхронно обновить `WebManager`, `DisplayDriver` и логику тревоги.

- [ ] Шаг 2: Убрать основные источники фрагментации heap и добавить контроль неуспешной инициализации.
  Категория: Работа с памятью.
  Что будет изменено: заменить частые передачи `String` по значению на `const String&` или фиксированные буферы; перевести `utf8ToWin1251()`, сборку QR payload, SSID/IP/URL и weather strings на предвыделенные буферы; заменить `NTPClient*` на обычный member-object; проверить результат `new`/`xQueueCreate()` и возвращать статус из `begin()` драйверов.
  Что это решит: уменьшит пиковое потребление RAM и деградацию памяти при долгой работе устройства, а также исключит падения на null dereference при нехватке памяти.
  Риски: возможны ошибки с длиной буферов, обрезание русских строк и побочные изменения в инициализации дисплея/аудио.

- [ ] Шаг 3: Перевести блокирующие участки запуска и reboot-flow на неблокирующую модель.
  Категория: Тайминги (`millis`).
  Что будет изменено: убрать длинные `delay()` из `NetworkManager::begin()`, `setupAP()`, web-handlers и audio-init там, где это можно заменить state machine или deferred action; перезагрузку после HTTP-ответа вынести в отложенный флаг, обрабатываемый из отдельного цикла/задачи; подключение Wi-Fi и fallback в AP перевести на поэтапную обработку по `millis()`.
  Что это решит: уменьшит зависания на старте, сократит риск пропуска watchdog/service loops и сделает web-интерфейс отзывчивее во время смены настроек.
  Риски: изменится порядок старта системы; возможны регрессии в AP fallback и в моменте, когда пользователь ожидает мгновенный `ESP.restart()`.

- [ ] Шаг 4: Ввести fail-safe поведение для критичных периферийных модулей.
  Категория: Работа с железом.
  Что будет изменено: `DisplayDriver::begin()`, `AudioDriver::begin()` и сетевую инициализацию сделать возвращающими статус; при сбое отключать только зависимый функционал, а не весь runtime; для очереди аудио добавить контроль переполнения и диагностический лог; для дисплея проверять создание canvas до первого использования.
  Что это решит: локализует отказы железа и предотвратит полное падение прошивки из-за одного проблемного модуля.
  Риски: появятся новые ветви деградации; потребуется проверить, что UI и alarm-логика корректно работают, когда часть модулей недоступна.

## 2. Priority 2: Performance & Hardware (Medium Impact)
(Оптимизация delay, работа с периферией, типы данных)

- [ ] Шаг 5: Нормализовать все циклические операции на `millis()` и выровнять частоты обновления задач.
  Категория: Тайминги (`millis`).
  Что будет изменено: унифицировать scheduler-паттерн для `TaskDisplay`, `TaskNetwork`, `loop()` и LED-анимаций; использовать безопасные wraparound-сравнения во всех таймерах; ограничить полные redraw дисплея и частоту тяжелых операций (`drawAPInfo`, weather redraw, reconnect attempts).
  Что это решит: снизит jitter, лишнюю загрузку CPU и вероятность того, что LED/web/display начнут мешать друг другу под нагрузкой.
  Риски: может измениться субъективная плавность анимаций и скорость реакции тревоги/интерфейса.

- [ ] Шаг 6: Сократить пиковое потребление RAM в HTTP/JSON цепочке и добавить валидацию входных данных.
  Категория: Работа с памятью.
  Что будет изменено: пересмотреть размеры `DynamicJsonDocument`, где возможно перейти на `StaticJsonDocument`; ограничить размер body у `/api/alarm/set`, `/api/settings/wifi`, `/api/settings/loc`; валидировать диапазоны времени, громкости, эффектов и яркости до записи в `Preferences`.
  Что это решит: уберет лишние аллокации при web-запросах, защитит от некорректных payload и исключит запись мусорных значений в persistent storage.
  Риски: web UI может начать отвергать payload, которые раньше неявно принимались; потребуется синхронизировать формат ответов с фронтендом.

- [ ] Шаг 7: Сверить программную конфигурацию с реальным железом и развести аппаратные профили.
  Категория: Работа с железом.
  Что будет изменено: проверить соответствие `config.h`, `README.md` и фактической сборки по дисплею, количеству WS2812, пинам фар/кнопки/UART; при расхождениях вынести аппаратные варианты в отдельные профили или compile-time presets; явно документировать ограничения ESP32-C3 по GPIO/UART/PWM.
  Что это решит: снизит риск "плавающих" дефектов, когда код логически исправен, но работает нестабильно из-за неверной распиновки или размеров дисплея/ленты.
  Риски: после правки профилей может поменяться ориентация экрана, индексы светодиодов и поведение подсветки на уже собранных устройствах.

## 3. Priority 3: Style & Clean Code (Low Impact)
(Нейминг, комментарии, структура)

- [ ] Шаг 8: Разделить orchestration-логику и драйверный код, чтобы следующие правки не конфликтовали между собой.
  Категория: Рефакторинг.
  Что будет изменено: вынести startup/reboot/state-management в отдельный слой или service-модули; сократить связность `Land_Cruiser.ino`; удалить дублирующиеся константы (`WDT_TIMEOUT` и `WDT_TIMEOUT_SEC`), унифицировать naming и привести API драйверов к одному стилю.
  Что это решит: упростит дальнейшие правки без каскада конфликтов и снизит цену сопровождения проекта.
  Риски: рефакторинг затронет много файлов сразу; высок шанс механических ошибок при переносе вызовов и include-зависимостей.

- [ ] Шаг 9: Почистить комментарии, кодировку, конфигурацию и эксплуатационную документацию.
  Категория: Рефакторинг.
  Что будет изменено: исправить mojibake в русских комментариях/README, вынести секреты (`WIFI_PASS`, API key) в шаблон-конфиг или локальный override-файл, обновить README под реальную архитектуру проекта (`WebServer` вместо устаревших описаний, фактический дисплей/LED count), добавить короткие комментарии только в нетривиальные места.
  Что это решит: улучшит читаемость, уменьшит риск утечки секретов и снимет путаницу между кодом, документацией и реальным железом.
  Риски: потребуется адаптировать локальную сборку и процесс прошивки, чтобы проект по-прежнему собирался без ручного поиска пропавших define'ов.

## 4. Verification Plan
(Как проверить, что после правок всё работает: тесты, мониторинг порта, замеры памяти).

- [ ] Сборка под `ESP32C3 Dev Module` с контролем размера прошивки, свободной RAM и предупреждений компилятора после каждого приоритета.
- [ ] Soak-test на 2-4 часа: Wi-Fi подключен, web UI опрашивает `/api/status`, LED-анимации активны, погода обновляется по таймеру, в Serial нет reset/WDT/panic.
- [ ] Проверка сценариев сети: первый старт без сохраненного Wi-Fi, переход в AP mode, сохранение новых credentials, reboot, повторное подключение после потери сети.
- [ ] Проверка дисплея и светодиодов: корректная ориентация экрана, отсутствие мерцания, правильные индексы фар/поворотников/подсветки, стабильная яркость.
- [ ] Проверка аудио: инициализация DFPlayer, проигрывание тестового трека, stop по кнопке, отсутствие зависаний очереди при серии команд из web UI.
- [ ] Замеры памяти и runtime: `ESP.getFreeHeap()`, largest free block, stack high-water mark для `TaskDisplay`, `TaskNetwork`, `TaskWeb`, а также число неуспешных аллокаций/ошибок очереди в debug-логах.
